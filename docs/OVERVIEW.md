## 一、项目背景与目标

### 1.1 项目定位

AI快讯是一个**多源聚合式 AI 新闻阅读器**，旨在为 AI 从业者和爱好者提供一站式的行业资讯获取体验。通过聚合 13+ 主流平台、160+ 信息源，结合自动化翻译和智能筛选，让用户在 5 分钟内掌握全球 AI 领域最新动态。

### 1.2 v2.0 迭代目标

从 v1.0 到 v2.0，项目围绕**"提升信息获取效率和阅读体验"**展开，核心目标：

1. **双语阅读**：解决英文资讯阅读门槛，支持中/英/双语三种模式
2. **信息沉淀**：通过收藏和历史记录，让有价值的内容可追溯
3. **交互体验**：优化长文阅读体验，提供更便捷的导航方式
4. **数据质量**：修复日期准确性问题，建立数据校验机制

## 二、架构决策回顾

### 2.1 技术选型复盘

| 技术选择 | 决策依据 | 实际表现 | 评价 |
|:---------|:---------|:---------|:----:|
| Vue 3 + Vite | 轻量、快速、生态成熟 | 开发效率高，构建速度快 | ✅ 优秀 |
| TailwindCSS | 原子化 CSS，开发效率高 | 样式一致性好，迭代快 | ✅ 优秀 |
| TypeScript | 类型安全，减少运行时错误 | 核心模块类型完整 | ✅ 良好 |
| Node.js 抓取脚本 | 同语言栈，开发维护成本低 | 运行稳定，扩展方便 | ✅ 优秀 |
| GitHub Actions | 免费 CI/CD，与 GitHub 深度集成 | 定时抓取稳定运行 | ✅ 优秀 |
| Vercel | 自动部署，全球 CDN | 部署体验极佳 | ✅ 优秀 |
| localStorage | 无服务端，零成本 | 满足当前需求 | ⚠️ 有局限 |

### 2.2 关键架构决策

#### 决策 1：静态 JSON + 前端渲染 vs 服务端 API

**选择：** 静态 JSON + 前端渲染

**理由：**
- 新闻数据更新频率不高（每 2 小时一次）
- 静态文件 CDN 缓存成本低
- 无服务端运维成本
- 适合个人项目规模

**复盘：** 正确的选择。项目上线以来运行稳定，访问速度快，运维成本几乎为零。

**潜在风险：** 数据量持续增长时，单次加载 JSON 可能过大。可考虑分页或按天分片。

---

#### 决策 2：Google 翻译 API 后端生成 vs 前端实时翻译

**选择：** 后端预生成双语字段

**理由：**
- 翻译结果持久化，避免重复调用 API
- 前端切换无延迟，体验更好
- 可控制翻译成本（限流 + 缓存）
- 不依赖前端网络环境

**复盘：** 正确的选择。前端翻译切换响应迅速，用户体验流畅。翻译成本通过缓存机制得到有效控制。

**优化空间：** 翻译质量评估机制、过期缓存清理策略。

---

#### 决策 3：Composables 模式 vs Pinia 状态管理

**选择：** Vue 3 Composables 模式

**理由：**
- 项目规模中等，不需要完整的状态管理库
- Composables 更轻量，组合灵活
- 逻辑复用性好
- 学习成本低

**复盘：** 适合当前阶段。各 Composable 职责清晰，依赖关系简单。

**未来考量：** 如果后续增加用户系统、云同步等功能，可考虑引入 Pinia 统一管理状态。

## 三、v2.0 新增功能架构

### 3.1 双语翻译系统

#### 架构设计

```
┌─────────────────────────────────────────────────────────┐
│                        后端                              │
│  ┌──────────┐    ┌──────────┐    ┌──────────────────┐  │
│  │ 数据抓取  │ →  │ 去重处理  │ →  │ Google 翻译 API  │  │
│  └──────────┘    └──────────┘    └────────┬─────────┘  │
│                                            │            │
│                               ┌────────────▼─────────┐  │
│                               │  title_zh_cache.json │  │
│                               └────────────┬─────────┘  │
│                                            │            │
│                               ┌────────────▼─────────┐  │
│                               │  生成双语 JSON 输出   │  │
│                               └────────────┬─────────┘  │
└────────────────────────────────────────────┼────────────┘
                                             │
┌────────────────────────────────────────────┼────────────┐
│                        前端                 │            │
│                               ┌────────────▼─────────┐  │
│                               │  useTranslate 状态    │  │
│                               │  translateMode ref   │  │
│                               └────────────┬─────────┘  │
│                                            │            │
│                               ┌────────────▼─────────┐  │
│                               │  getDisplayTitle()    │  │
│                               │  computed 计算属性    │  │
│                               └────────────┬─────────┘  │
│                                            │            │
│                               ┌────────────▼─────────┐  │
│                               │  NewsCard 标题渲染    │  │
│                               └──────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

#### 设计亮点

1. **后端预计算**：翻译在数据抓取阶段完成，前端零延迟切换
2. **缓存优先**：三级缓存策略（同 URL 中文标题 → 历史缓存 → API 调用）
3. **响应式切换**：基于 Vue computed，模式切换后所有标题自动更新
4. **优雅降级**：任一翻译字段缺失时，自动降级到可用字段

### 3.2 收藏与历史系统

#### 架构设计

```
useStorage.ts
├── 收藏模块
│   ├── favoritesRef (响应式状态)
│   ├── toggleFavorite()
│   ├── getNewFavoritesCount()
│   └── markFavoritesSeen()
├── 历史模块
│   ├── historyRef (响应式状态)
│   ├── addToHistory()
│   ├── getNewHistoryCount()
│   └── markHistorySeen()
└── 偏好模块
    ├── preferencesRef (响应式状态)
    ├── savePreferences()
    └── 主题 / 翻译模式 / 来源筛选
```

#### 新增计数设计

采用「上次查看数量」差值计算法：

```
新增数 = 当前总数 - lastSeenCount

打开弹窗时：
  lastSeenCount = 当前总数
  新增数 = 0
```

**优点：**
- 实现简单，无需额外事件日志
- 关闭页面重新打开仍然有效
- 数据持久化，刷新不丢失

### 3.3 滚动进度条增强

#### 交互设计

```
            ┌─────────────────┐
            │  ⬆  顶部按钮    │  ← 点击回顶部
            ├─────────────────┤
            │                 │
            │   进度填充条    │  ← 显示当前进度
            │                 │
            │   50% 文字      │  ← 百分比显示
            │                 │
            │   可点击/拖拽   │  ← 交互区域
            │                 │
            ├─────────────────┤
            │  ⬇  底部按钮    │  ← 点击跳到底部
            └─────────────────┘
```

#### 技术要点

1. **点击定位**：通过 `getBoundingClientRect()` 计算点击位置百分比
2. **拖拽实现**：mousedown → mousemove → mouseup 事件链，拖拽时禁用 transition
3. **滚动同步**：双向绑定，滚动更新进度条，拖拽更新滚动位置
4. **智能显隐**：顶部隐藏，滚动后显示，避免干扰首屏

### 3.4 日期校验系统

#### 问题根因分析

NewsNow 等数据源的 `published_at` 字段可能返回未来日期，原因：
- 数据源时区配置错误
- 文章预发布时间标记
- 爬虫解析异常

#### 修复策略

**1 小时宽容度原则：**
- 发布时间 - 当前时间 ≤ 1 小时：视为正常（时区差异、时钟偏差）
- 发布时间 - 当前时间 > 1 小时：视为无效，替换为抓取时间

**双层校验：**
- 新抓取数据：抓取时即时校验
- 历史归档：每次运行时批量修复

## 五、经验与教训

### 5.1 成功经验

1. **静态优先原则**：能静态生成的就不要走服务端，运维成本差异巨大
2. **缓存前置思维**：翻译、去重等计算尽量在数据端完成，前端只负责展示
3. **渐进式增强**：核心功能先上线，体验优化迭代推进
4. **Composables 模式**：Vue 3 的组合式 API 在中等规模项目中非常高效

### 5.2 教训总结

1. **数据质量不可轻信**：第三方数据源的字段不一定可靠，必须有校验机制
2. **暗黑模式设计**：纯文字按钮在暗黑模式下可能不可见，优先用图标/Emoji
3. **用户数据持久化**：涉及用户操作的功能，一定要考虑刷新后的状态恢复
4. **交互反馈重要性**：新增计数、动画过渡等细节，显著提升用户体验感知

| 维度 | 评分 (1-10) | 说明 |

| 稳定性 | 9 | 上线以来运行稳定 |
| 安全性 | 8 | 无服务端，攻击面小 |
| 测试覆盖 | 5 | 单元测试不足 |
| 文档完整度 | 9 | 文档体系完善 |

**综合评分：8.0 / 10** - 优秀

### 6.2 核心优势

1. **极低运维成本**：全静态托管，几乎零运维
2. **良好的扩展性**：模块化设计，新增数据源成本低

4. **完善的文档体系**：PRD → ARCH → SDD → TEST → DEPLOY 全链路文档

1. **数据量增长**：JSON 文件持续增大可能影响首屏加载
2. **存储限制**：localStorage 容量上限可能影响收藏/历史功能
3. **翻译依赖**：Google 翻译 API 可用性依赖网络环境
4. **测试薄弱**：缺乏自动化测试，回归风险较高

## 七、总结

AI快讯 v2.0 是一次成功的迭代，在 v1.0 基础上新增了双语翻译、收藏历史、滚动增强等重要功能，同时修复了数据质量问题。架构选择经受住了实践检验，静态化 + 预计算的策略在个人项目规模下表现优异。

未来的演进方向应聚焦于：
- **质量建设**：补充测试，提升代码健壮性
- **性能兜底**：解决存储容量、大数据量等边界问题
- **体验深化**：在交互细节和个性化方面持续优化
